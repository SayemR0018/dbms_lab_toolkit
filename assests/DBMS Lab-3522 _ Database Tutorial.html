Sample Database Tables

All examples throughout this tutorial use the following two tables: Students and Courses. These tables represent a simple university database. Create them before running any example.

MS SQL Server — Create Tables 
-- Create Students Table
CREATE TABLE Students (
  StudentID   INT PRIMARY KEY IDENTITY(1,1),
  FullName    NVARCHAR(100) NOT NULL,
  Department  NVARCHAR(50),
  CGPA        DECIMAL(3,2),
  EnrollYear  INT,
  IsActive    BIT DEFAULT 1
);

-- Create Courses Table
CREATE TABLE Courses (
  CourseID    INT PRIMARY KEY IDENTITY(1,1),
  CourseName  NVARCHAR(100),
  Credits     INT,
  Department  NVARCHAR(50),
  StudentID   INT FOREIGN KEY
              REFERENCES Students(StudentID)
);
MySQL — Create Tables 
-- Create Students Table
CREATE TABLE Students (
  StudentID   INT PRIMARY KEY AUTO_INCREMENT,
  FullName    VARCHAR(100) NOT NULL,
  Department  VARCHAR(50),
  CGPA        DECIMAL(3,2),
  EnrollYear  INT,
  IsActive    TINYINT(1) DEFAULT 1
);

-- Create Courses Table
CREATE TABLE Courses (
  CourseID    INT PRIMARY KEY AUTO_INCREMENT,
  CourseName  VARCHAR(100),
  Credits     INT,
  Department  VARCHAR(50),
  StudentID   INT,
  FOREIGN KEY (StudentID)
    REFERENCES Students(StudentID)
);

Database Views

A View is a virtual table based on the result of a SQL query. It does not store data physically — it pulls data from the underlying base tables each time it is queried. Views are used to simplify complex queries, restrict access to sensitive columns, and present data in a customized format.

📌 Key Characteristics
  • A view acts like a table but stores no actual data — only the query definition.
  • Changes in base tables are automatically reflected when the view is queried.
  • Views can be used to implement row-level and column-level security.
  • Complex JOINs and aggregations can be hidden behind a simple view name.
  • Some views are updatable (INSERT/UPDATE/DELETE) if they meet certain conditions.
Syntax: CREATE VIEW view_name AS SELECT ... — The SELECT can include JOINs, WHERE clauses, aggregates, and expressions.
Example 1 — Simple View

Create a view that shows only active CSE students with their CGPA.

MS SQL Server
-- Create View
CREATE VIEW vw_CSE_Students AS
SELECT
  StudentID,
  FullName,
  CGPA,
  EnrollYear
FROM Students
WHERE Department = 'CSE'
  AND IsActive = 1;

-- Query the View
SELECT * FROM vw_CSE_Students;

-- Drop View
DROP VIEW vw_CSE_Students;
MySQL

Example 2 — JOIN View

Create a view that joins Students and Courses to show enrollment details.

MS SQL Server
CREATE VIEW vw_StudentCourses AS
SELECT
  s.StudentID,
  s.FullName,
  s.Department,
  c.CourseName,
  c.Credits
FROM Students s
INNER JOIN Courses c
  ON s.StudentID = c.StudentID

-- Use the View

// 02 · Section Two

Functions

A Function in SQL is a reusable block of code that accepts input parameters, performs a computation, and returns a value. Functions are used inside SELECT, WHERE, and other SQL clauses. There are two main categories: System (Built-in) Functions provided by the database engine, and User-Defined Functions (UDF) created by the developer.

Key Rule: A function MUST return a value (scalar or table). It cannot perform DML operations (INSERT/UPDATE/DELETE on real tables in most databases).
🏭
System Functions
Built into the DBMS. Examples: LEN(), UPPER(), GETDATE(), ROUND(), COUNT(), SUM(), etc.
🛠
Scalar UDF
Returns a single value (INT, VARCHAR, etc.). Created with CREATE FUNCTION.
📊
Table-Valued UDF
Returns a table result set. Can be used in FROM clause like a table.
// 02a · System Functions

System Functions

System functions are pre-built functions provided by the database engine. They are ready to use without any definition. They cover string manipulation, date/time operations, numeric calculations, and aggregate operations.

String Functions
MS SQL Server
-- String System Functions
SELECT
  FullName,
  LEN(FullName)          AS NameLength,
  UPPER(FullName)        AS UpperName,
  LOWER(FullName)        AS LowerName,
  LEFT(FullName, 5)      AS First5Chars,
  SUBSTRING(FullName,
    1, CHARINDEX(' ',FullName)-1)
                         AS FirstName,
  LTRIM(RTRIM(FullName)) AS TrimmedName,
  REPLACE(Department,
    'CSE','Computer Sci') AS DeptFull
FROM Students;
MySQL
-- String System Functions
SELECT
  FullName,
  LENGTH(FullName)        AS NameLength,
  UPPER(FullName)         AS UpperName,
  LOWER(FullName)         AS LowerName,
  LEFT(FullName, 5)       AS First5Chars,
  SUBSTRING(FullName,
    1, LOCATE(' ',FullName)-1)
                          AS FirstName,
  TRIM(FullName)          AS TrimmedName,
  REPLACE(Department,
    'CSE','Computer Sci')  AS DeptFull
FROM Students;
Date & Numeric Functions
MS SQL Server
-- Date Functions
SELECT
  FullName,

// 02b · User-Defined Functions

User-Defined Functions (UDF)

User-Defined Functions are custom functions written by the programmer to encapsulate business logic that can be reused inside SQL queries. There are two main types: Scalar Functions (return one value) and Table-Valued Functions (return a table).

Scalar UDF — Grade Calculator

A scalar function that accepts a CGPA value and returns the corresponding letter grade.

MS SQL Server
-- Create Scalar UDF
CREATE FUNCTION dbo.GetGrade(
  @CGPA DECIMAL(3,2)
)
RETURNS VARCHAR(10)
AS
BEGIN
  DECLARE @Grade VARCHAR(10)
  IF @CGPA >= 3.75
    SET @Grade = 'A+'
  ELSE IF @CGPA >= 3.50
    SET @Grade = 'A'
  ELSE IF @CGPA >= 3.00
    SET @Grade = 'B+'
  ELSE
    SET @Grade = 'B'
  RETURN @Grade
END;

-- Call the function in SELECT
SELECT
  FullName,
  CGPA,
  dbo.GetGrade(CGPA) AS Grade
FROM Students;
MySQL
-- Delimiter change required in MySQL
DELIMITER $$

CREATE FUNCTION GetGrade(
  p_CGPA DECIMAL(3,2)
)
RETURNS VARCHAR(10)
DETERMINISTIC
BEGIN
  DECLARE v_Grade VARCHAR(10);
  IF p_CGPA >= 3.75 THEN
    SET v_Grade = 'A+';
  ELSEIF p_CGPA >= 3.50 THEN
    SET v_Grade = 'A';
  ELSEIF p_CGPA >= 3.00 THEN
    SET v_Grade = 'B+';
  ELSE
    SET v_Grade = 'B';
  END IF;
  RETURN v_Grade;
END$$
DELIMITER ;

-- Call the function
SELECT
  FullName,
  CGPA,
  GetGrade(CGPA) AS Grade
FROM Students;

Table-Valued UDF — Department Filter

A table-valued function that accepts a department name and returns the filtered students as a table result.

MS SQL Server — Inline TVF
CREATE FUNCTION
  dbo.GetStudentsByDept(
    @Dept NVARCHAR(50)
  )
RETURNS TABLE
AS
RETURN
(
  SELECT
    StudentID,
    FullName,
    CGPA,
    EnrollYear
  FROM Students
  WHERE Department = @Dept
    AND IsActive = 1
);

-- Use like a table in FROM
SELECT *
FROM dbo.GetStudentsByDept('CSE')
ORDER BY CGPA DESC;
MySQL — (Use Stored Proc / View)
-- MySQL does NOT support
-- table-valued functions directly.
-- Use a Stored Procedure instead:

DELIMITER $$
CREATE PROCEDURE
  GetStudentsByDept(
    IN p_Dept VARCHAR(50)
  )
BEGIN
  SELECT
    StudentID,
    FullName,
    CGPA,
    EnrollYear
  FROM Students
  WHERE Department = p_Dept
    AND IsActive = 1
  ORDER BY CGPA DESC;
END$$
DELIMITER ;

-- Call it
CALL GetStudentsByDept('CSE');
Note for MySQL: MySQL does not support Table-Valued Functions. Use a Stored Procedure with a SELECT statement instead for returning table results.
// 03 · Section Three

Stored Procedures

A Stored Procedure is a precompiled, named block of SQL statements stored in the database and executed by calling its name. Unlike functions, procedures do not need to return a value — they can perform INSERT, UPDATE, DELETE, SELECT, and complex business logic including transactions, error handling, and loops.

📌 Key Characteristics
  • Stored procedures are precompiled — execution plan is cached for better performance.
  • They can have IN (input), OUT (output), and INOUT parameters.
  • Can perform full DML operations: INSERT, UPDATE, DELETE.
  • Support transaction management (BEGIN TRAN / COMMIT / ROLLBACK).
  • Can call other procedures and functions within them.
  • Improves security by limiting direct table access — users call procedures instead.
Example 1 — Basic SELECT Procedure
MS SQL Server
-- Create Stored Procedure
CREATE PROCEDURE
  usp_GetActiveStudents
AS
BEGIN
  SELECT
    StudentID,
    FullName,
    Department,
    CGPA
  FROM Students
  WHERE IsActive = 1
  ORDER BY CGPA DESC;
END;

-- Execute the Procedure
EXEC usp_GetActiveStudents;
MySQL
-- Create Stored Procedure
DELIMITER $$
CREATE PROCEDURE
  usp_GetActiveStudents()
BEGIN
  SELECT
    StudentID,
    FullName,
    Department,
    CGPA
  FROM Students
  WHERE IsActive = 1
  ORDER BY CGPA DESC;
END$$
DELIMITER ;

-- Execute the Procedure
CALL usp_GetActiveStudents();
Example 2 — Procedure with Parameters

A procedure that inserts a new student and returns the new ID.

MS SQL Server — Input & Output Params
CREATE PROCEDURE usp_AddStudent
  @Name       NVARCHAR(100),
  @Dept       NVARCHAR(50),
  @CGPA       DECIMAL(3,2),
  @Year       INT,
  @NewID      INT OUTPUT
AS
BEGIN
  INSERT INTO Students
    (FullName, Department, CGPA,
     EnrollYear, IsActive)
  VALUES
    (@Name, @Dept, @CGPA,
     @Year, 1);

  SET @NewID = SCOPE_IDENTITY();
END;

-- Execute with Output Param
DECLARE @InsertedID INT;
EXEC usp_AddStudent
  'Karim Uddin', 'CSE',
  3.55, 2024,
  @NewID = @InsertedID OUTPUT;
PRINT 'New Student ID: '
      + CAST(@InsertedID AS VARCHAR);
MySQL — IN and OUT Params
DELIMITER $$
CREATE PROCEDURE usp_AddStudent(
  IN  p_Name   VARCHAR(100),
  IN  p_Dept   VARCHAR(50),
  IN  p_CGPA   DECIMAL(3,2),
  IN  p_Year   INT,
  OUT p_NewID  INT
)
BEGIN
  INSERT INTO Students
    (FullName, Department, CGPA,
     EnrollYear, IsActive)
  VALUES
    (p_Name, p_Dept, p_CGPA,
     p_Year, 1);

  SET p_NewID = LAST_INSERT_ID();
END$$
DELIMITER ;

-- Execute with Output Param
CALL usp_AddStudent(
  'Karim Uddin', 'CSE',
  3.55, 2024, @NewID
);
SELECT @NewID AS InsertedStudentID;
Example 3 — Procedure with Transaction & Error Handling
MS SQL Server — TRY/CATCH
CREATE PROCEDURE
  usp_UpdateCGPA
  @StudentID  INT,
  @NewCGPA    DECIMAL(3,2)
AS
BEGIN
  BEGIN TRY
    BEGIN TRANSACTION;

    UPDATE Students
    SET    CGPA = @NewCGPA
    WHERE  StudentID = @StudentID;

    COMMIT TRANSACTION;
    PRINT 'CGPA updated successfully.';
  END TRY
  BEGIN CATCH
    ROLLBACK TRANSACTION;
    PRINT 'Error: ' +
      ERROR_MESSAGE();
  END CATCH
END;

-- Call it
EXEC usp_UpdateCGPA 3, 3.20;
MySQL — DECLARE HANDLER
DELIMITER $$
CREATE PROCEDURE
  usp_UpdateCGPA(
  IN p_StudentID  INT,
  IN p_NewCGPA    DECIMAL(3,2)
)
BEGIN
  DECLARE EXIT HANDLER
    FOR SQLEXCEPTION
  BEGIN
    ROLLBACK;
    SELECT 'Error occurred'
           AS Message;
  END;

  START TRANSACTION;

  UPDATE Students
  SET    CGPA = p_NewCGPA
  WHERE  StudentID = p_StudentID;

  COMMIT;
  SELECT 'CGPA updated.' AS Message;
END$$
DELIMITER ;

-- Call it
CALL usp_UpdateCGPA(3, 3.20);
Best Practice: Always wrap data-modifying procedures in transactions. Use TRY/CATCH (SQL Server) or DECLARE EXIT HANDLER (MySQL) to rollback changes if any error occurs mid-execution.

View vs Function

Both Views and Functions can be used in SELECT queries, but they serve different purposes and have different capabilities.

Aspect🔵 VIEW🟡 FUNCTION
DefinitionVirtual table based on a stored SELECT queryNamed reusable logic that returns a value or table
ReturnsAlways returns a table (result set)Can return a scalar value OR a table
ParametersCannot accept parametersCan accept input parameters
Usage in SQLUsed in FROM clause like a tableUsed in SELECT, WHERE, FROM (TVF) clauses
DML OperationsSome simple views allow INSERT/UPDATECannot perform DML on real tables (MSSQL/MySQL)
PrecompiledNot precompiled (query runs each time)Not precompiled (similar to view)
WHERE filterCannot be parameterized; filter from outsideParameter acts as dynamic WHERE internally
ReusabilityReusable as a named query shortcutReusable callable unit of business logic
ComplexityGood for fixed presentation logicGood for conditional, calculated results
Created withCREATE VIEWCREATE FUNCTION
Called withSELECT * FROM view_nameSELECT fn() or SELECT * FROM tvf()
Rule of Thumb: Use a View when you want to simplify a fixed, often-used query. Use a Function when the result depends on dynamic input parameters or requires conditional logic.
// 05 · Comparison

Function vs Stored Procedure

Functions and Stored Procedures are both reusable programming objects, but they differ significantly in usage, capabilities, and when you should use one over the other.

Aspect🟡 FUNCTION🟢 STORED PROCEDURE
Must Return ValueYes — MUST return a value (scalar or table)No — can return 0 or more result sets
Return MechanismRETURN statement returns a valueSELECT or OUTPUT parameters return data
DML OperationsCannot modify real tables (generally)Can perform INSERT, UPDATE, DELETE freely
TransactionsCannot manage transactionsFull transaction control (BEGIN/COMMIT/ROLLBACK)
Error HandlingLimited error handlingFull TRY/CATCH or HANDLER block support
Called In SELECTYes — can be used inside SELECT, WHERENo — cannot be called inside SELECT
Called WithSELECT fn() or FROM tvf()EXEC proc / CALL proc()
ParametersIN parameters onlyIN, OUT, INOUT parameters supported
PrecompiledNot cached in same wayExecution plan cached — better for heavy operations
Use CaseCalculations, transformations, lookupsBusiness operations, batch processing, workflows
Call Another ProcCannot call a stored procedureCan call other procedures and functions
Created WithCREATE FUNCTIONCREATE PROCEDURE / CREATE PROC
Decision Guide: If you need to calculate something and use the result inside a query → use a Function. If you need to execute a business operation that modifies data, sends results, or needs transactions → use a Stored Procedure.
Quick Summary Tags
VIEW = Virtual Table VIEW = No Parameters FUNCTION = Returns Value FUNCTION = Used in SELECT FUNCTION = No DML PROCEDURE = Full DML PROCEDURE = Transaction Support PROCEDURE = IN/OUT Params PROCEDURE = Not in SELECT
When to Use — Quick Cheat Sheet
👁
Use VIEW when
You want to simplify a complex recurring query. Hide sensitive columns. Provide a consistent data interface to multiple users. No dynamic filtering needed.
⚙
Use FUNCTION when
You need a reusable calculation or lookup that accepts input and returns a value. It should be callable inline inside SQL expressions.
📦
Use PROCEDURE when
You need to execute a multi-step database operation — inserts, updates, deletes — especially with transactions, output params, or error handling.
