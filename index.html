🔌 Method 1 — Connect Using MySQLi

MySQLi (MySQL Improved) is the procedural/OOP extension built into PHP. Fastest and simplest for MySQL-only projects.

PHP — MySQLi Procedural 
<?php
// ── Database credentials ──
$host   = "localhost";
$user   = "root";
$pass   = "";         // XAMPP default: empty
$dbname = "testdb";

// ── Create connection ──
$conn = mysqli_connect($host, $user, $pass, $dbname);

// ── Check connection ──
if (!$conn) {
    die("❌ Connection failed: " . mysqli_connect_error());
}

echo "✅ Connected successfully to MySQL!";

// ── Close connection ──
mysqli_close($conn);
?>
✅ Connected successfully to MySQL!
❌ Connection failed: Access denied for user 'root'@'localhost'
04
🎯 Method 2 — Connect Using MySQLi OOP

Object-Oriented approach is cleaner and preferred for modern PHP development.

PHP — MySQLi Object-Oriented

$conn = new mysqli("localhost", "root", "", "testdb");

if ($conn->connect_error) {
    die("❌ Failed: " . $conn->connect_error);

05
🛡️ Method 3 — Connect Using PDO (Recommended)

PDO (PHP Data Objects) is the most secure and flexible method. It supports multiple database drivers and prepared statements natively.

PHP — PDO MySQL 
<?php
$host   = "localhost";
$dbname = "testdb";
$user   = "root";
$pass   = "";

try {
    $dsn  = "mysql:host=$host;dbname=$dbname;charset=utf8mb4";
    $pdo  = new PDO($dsn, $user, $pass);

    // Set error mode to exception
    $pdo->setAttribute(PDO::ATTR_ERRMODE,
                     PDO::ERRMODE_EXCEPTION);

    echo "✅ PDO Connected to MySQL successfully!";

} catch (PDOException $e) {
    echo "❌ Connection failed: " . $e->getMessage();
}
?>
💡 PDO supports MySQL, SQL Server, PostgreSQL, SQLite and more — switch databases by just changing the DSN string.

🔌 Method 1 — Connect Using sqlsrv_connect()

The native Microsoft sqlsrv extension provides sqlsrv_connect() for direct SQL Server connections.

PHP — sqlsrv Extension 
<?php
// ── Server and credentials ──
$serverName = "LPT5032";       // or ".\SQLEXPRESS"
$connOptions = [
    "Database"  => "aplbanking",
    "Uid"       => "sa",
    "PWD"       => "Bitopi@1234"
];

// ── Attempt connection ──
$conn = sqlsrv_connect($serverName, $connOptions);

if ($conn) {
    echo "✅ Connected to SQL Server successfully!";
    sqlsrv_close($conn);
} else {
    echo "❌ Connection FAILED!<br>";
    foreach (sqlsrv_errors() as $err) {
        echo "Code: "    . $err['code']    . "<br>";
        echo "Message: " . $err['message'] . "<br>";
    }
}
?>
✅ Connected to SQL Server successfully!

🌿 Method 2 — Entity Framework Core (Code-First)

C# — EF Core with MySQL (Pomelo)

// Install-Package Pomelo.EntityFrameworkCore.MySql

using Microsoft.EntityFrameworkCore;

// ── Model ──
public class User
{
    public int    Id    { get; set; }
    public string Name  { get; set; }
    public string Email { get; set; }
}

// ── DbContext ──
public class AppDbContext : DbContext
{

}

// ── Usage ──
using var db = new AppDbContext();
var users = db.Users.ToList();
users.ForEach(u => Console.WriteLine($"{u.Name} — {u.Email}"));

⚙️ Enable SQL Server for Remote/Network Access

If Visual Studio is on a different machine than SQL Server, you must enable TCP/IP and the right authentication mode.

  • Open SQL Server Configuration Manager
  • Go to SQL Server Network Configuration → Protocols for MSSQLSERVER
  • Enable TCP/IP → Right-click → Enable → Restart SQL Server service
  • In SSMS: Right-click Server → Properties → Security → Select "SQL Server and Windows Authentication mode"
  • Enable sa login: Security → Logins → sa → Properties → Status → Login: Enabled
⚠️ After changing authentication mode or enabling SA, you must restart the SQL Server service for changes to take effect.

Database Views

A View is a virtual table based on the result of a SQL query. It does not store data physically — it pulls data from the underlying base tables each time it is queried. Views are used to simplify complex queries, restrict access to sensitive columns, and present data in a customized format.

📌 Key Characteristics
  • A view acts like a table but stores no actual data — only the query definition.
  • Changes in base tables are automatically reflected when the view is queried.
  • Views can be used to implement row-level and column-level security.
  • Complex JOINs and aggregations can be hidden behind a simple view name.
  • Some views are updatable (INSERT/UPDATE/DELETE) if they meet certain conditions.
Syntax: CREATE VIEW view_name AS SELECT ... — The SELECT can include JOINs, WHERE clauses, aggregates, and expressions.
Example 1 — Simple View

Create a view that shows only active CSE students with their CGPA.

  EnrollYear

-- Drop View

Functions

A Function in SQL is a reusable block of code that accepts input parameters, performs a computation, and returns a value. Functions are used inside SELECT, WHERE, and other SQL clauses. There are two main categories: System (Built-in) Functions provided by the database engine, and User-Defined Functions (UDF) created by the developer.

Key Rule: A function MUST return a value (scalar or table). It cannot perform DML operations (INSERT/UPDATE/DELETE on real tables in most databases).
🏭
System Functions
Built into the DBMS. Examples: LEN(), UPPER(), GETDATE(), ROUND(), COUNT(), SUM(), etc.
🛠
Scalar UDF
Returns a single value (INT, VARCHAR, etc.). Created with CREATE FUNCTION.
📊
Table-Valued UDF
Returns a table result set. Can be used in FROM clause like a table.

User-Defined Functions (UDF)

User-Defined Functions are custom functions written by the programmer to encapsulate business logic that can be reused inside SQL queries. There are two main types: Scalar Functions (return one value) and Table-Valued Functions (return a table).

Scalar UDF — Grade Calculator

A scalar function that accepts a CGPA value and returns the corresponding letter grade.

MS SQL Server
-- Create Scalar UDF
CREATE FUNCTION dbo.GetGrade(
  @CGPA DECIMAL(3,2)
)
RETURNS VARCHAR(10)
AS
BEGIN
  DECLARE @Grade VARCHAR(10)
  IF @CGPA >= 3.75
    SET @Grade = 'A+'
  ELSE IF @CGPA >= 3.50
    SET @Grade = 'A'
  ELSE IF @CGPA >= 3.00
    SET @Grade = 'B+'
  ELSE
    SET @Grade = 'B'
  RETURN @Grade
END;

-- Call the function in SELECT
SELECT
  FullName,
  CGPA,
  dbo.GetGrade(CGPA) AS Grade
FROM Students;

// 03 · Section Three

Stored Procedures

A Stored Procedure is a precompiled, named block of SQL statements stored in the database and executed by calling its name. Unlike functions, procedures do not need to return a value — they can perform INSERT, UPDATE, DELETE, SELECT, and complex business logic including transactions, error handling, and loops.

📌 Key Characteristics
  • Stored procedures are precompiled — execution plan is cached for better performance.
  • They can have IN (input), OUT (output), and INOUT parameters.
  • Can perform full DML operations: INSERT, UPDATE, DELETE.
  • Support transaction management (BEGIN TRAN / COMMIT / ROLLBACK).
  • Can call other procedures and functions within them.
  • Improves security by limiting direct table access — users call procedures instead.
Example 1 — Basic SELECT Procedure
MS SQL Server
-- Create Stored Procedure

  usp_GetActiveStudents

BEGIN

    StudentID,

  WHERE IsActive = 1

Example 3 — Procedure with Transaction & Error Handling
MS SQL Server — TRY/CATCH
CREATE PROCEDURE
  usp_UpdateCGPA
  @StudentID  INT,
  @NewCGPA    DECIMAL(3,2)
AS
BEGIN
  BEGIN TRY
    BEGIN TRANSACTION;

    UPDATE Students
    SET    CGPA = @NewCGPA
    WHERE  StudentID = @StudentID;

    COMMIT TRANSACTION;
    PRINT 'CGPA updated successfully.';
  END TRY
  BEGIN CATCH
    ROLLBACK TRANSACTION;
    PRINT 'Error: ' +
      ERROR_MESSAGE();
  END CATCH
END;

-- Call it

MySQL — DECLARE HANDLER
DELIMITER $$
CREATE PROCEDURE
  usp_UpdateCGPA(
  IN p_StudentID  INT,
  IN p_NewCGPA    DECIMAL(3,2)
)
BEGIN
  DECLARE EXIT HANDLER
    FOR SQLEXCEPTION
  BEGIN
    ROLLBACK;
    SELECT 'Error occurred'
           AS Message;
  END;

  START TRANSACTION;

  UPDATE Students
  SET    CGPA = p_NewCGPA
  WHERE  StudentID = p_StudentID;

02
FK Foreign Key Constraint
Enforces referential integrity between two tables

A Foreign Key links a column (or set of columns) in one table to the Primary Key of another table. It prevents orphan records — you cannot insert a child row that doesn't match a parent, and you cannot delete a parent that has children (unless ON DELETE CASCADE is set).

MS SQL Server
CREATE TABLE Departments (
  DeptID   INT           PRIMARY KEY
                            IDENTITY(1,1),
  DeptName NVARCHAR(100) NOT NULL
);

CREATE TABLE Employees (
  EmpID    INT           PRIMARY KEY
                            IDENTITY(1,1),
  EmpName  NVARCHAR(100) NOT NULL,
  DeptID   INT,
  CONSTRAINT FK_Emp_Dept
    FOREIGN KEY (DeptID)
    REFERENCES Departments(DeptID)
    ON DELETE CASCADE
    ON UPDATE CASCADE
);

A Primary Key (or Unique key) made of two or more columns

A Composite Key uses multiple columns together to uniquely identify a row. Common in junction/bridge tables that represent Many-to-Many relationships (e.g., a student can enroll in many courses, and a course can have many students — the combination of StudentID + CourseID is unique).

MS SQL Server — Enrollment (Junction Table)
CREATE TABLE Enrollment (
  StudentID INT  NOT NULL,
  CourseID  INT  NOT NULL,
  EnrollDate DATE DEFAULT GETDATE(),
  Grade     CHAR(2),

  -- Composite Primary Key
  CONSTRAINT PK_Enrollment
    PRIMARY KEY (StudentID, CourseID),

  CONSTRAINT FK_Enr_Student
    FOREIGN KEY (StudentID)
    REFERENCES Students(StudentID),

  CONSTRAINT FK_Enr_Course
    FOREIGN KEY (CourseID)
    REFERENCES Courses(CourseID)
);

NOT NULL Constraint
Forces a column to always have a value — empty/missing data is not allowed

The NOT NULL constraint prevents a column from holding a NULL value. It's a column-level constraint and can be applied at creation or via ALTER TABLE. Columns without NOT NULL default to NULLable. In SQL Server, use NVARCHAR for Unicode; MySQL uses VARCHAR.

MS SQL Server
CREATE TABLE Products (
  ProductID    INT
                  PRIMARY KEY
                  IDENTITY(1,1),
  ProductName  NVARCHAR(200) NOT NULL,
  SKU          NVARCHAR(50)  NOT NULL,
  Price        DECIMAL(10,2) NOT NULL,
  Description NVARCHAR(MAX) -- NULLable
);

-- Add NOT NULL to existing column
ALTER TABLE Products
ALTER COLUMN SKU NVARCHAR(50) NOT NULL;

Automatically fills a column with a preset value when no value is provided

The DEFAULT constraint assigns a fallback value to a column when an INSERT statement does not supply a value. Defaults can be literals (e.g., 0, 'Active') or functions (e.g., current timestamp). This is widely used for audit columns like CreatedAt, IsActive, status fields, etc.

    DEFAULT 1,

07
CHK Check Constraint
Validates that column values satisfy a specified logical condition

A CHECK constraint lets you define a boolean expression that every row's value must satisfy. If an INSERT or UPDATE would cause the expression to evaluate to FALSE, the operation is rejected. Useful for business rules like age limits, price ranges, status enumerations, and date logic.

MS SQL Server
CREATE TABLE Employees (
  EmpID      INT           PRIMARY KEY
                             IDENTITY(1,1),

  Age        INT,
  Salary     DECIMAL(12,2),
  Gender     CHAR(1),
  JoinDate   DATE,

  CONSTRAINT CHK_Emp_Age
    CHECK (Age BETWEEN 18 AND 65),

  CONSTRAINT CHK_Emp_Salary
    CHECK (Salary >= 5000),

  CONSTRAINT CHK_Emp_Gender
    CHECK (Gender IN ('M', 'F', 'O')),

  CONSTRAINT CHK_Emp_JoinDate
    CHECK (JoinDate >= '2000-01-01')
);

Student Transcript — MS SQL Server
SELECT
  s.StudentName,
  d.DeptName,
  c.CourseName,
  c.CourseCode,
  e.Semester,
  e.Grade,
  SUM(ex.Marks) AS TotalMarks
FROM Students          s
JOIN Departments        d
  ON s.DeptID    = d.DeptID
JOIN Enrollment         e
  ON s.StudentID = e.StudentID
JOIN Courses             c
  ON e.CourseID  = c.CourseID
LEFT JOIN Exams          ex
  ON ex.StudentID = s.StudentID
  AND ex.CourseID = c.CourseID
WHERE s.StudentID = 1
GROUP BY
  s.StudentName, d.DeptName,
  c.CourseName,  c.CourseCode,
  e.Semester,    e.Grade;

SELECT

    DeptID,

-- Dept-level salary stats
SELECT
    d.DeptName,
    COUNT(e.EmpID) AS HeadCount,
    SUM(e.Salary)  AS TotalSalary,

-- Average salary rounded to 2 decimals
SELECT
    d.DeptName,
    ROUND(AVG(e.Salary), 2) AS AvgSalary
FROM       Employees   e
INNER JOIN Departments d ON e.DeptID = d.DeptID

🔵 SQL Server: ROUND(expr, decimals). Use FORMAT(val, 'N2') for display formatting with commas.
MySQL — ROUND & FORMAT
-- Average salary rounded to 2 decimals
SELECT
    d.DeptName,
    ROUND(AVG(e.Salary), 2) AS AvgSalary
FROM       Employees   e
INNER JOIN Departments d ON e.DeptID = d.DeptID

Post-Aggregate Filter

⚡
WHERE vs HAVING
Combining both in one query for precise control
Key Difference
WHERE filters rows first, HAVING filters groups — both DBs
-- WHERE: only include joined-before-2022 employees
-- HAVING: only show depts with >= 2 such employees
SELECT
    d.DeptName,
    COUNT(e.EmpID)  AS EmpCount,
    AVG(e.Salary)   AS AvgSalary
FROM       Employees   e
INNER JOIN Departments d ON e.DeptID = d.DeptID
WHERE      e.JoinDate < '2022-01-01'
GROUP BY   d.DeptName
HAVING     COUNT(e.EmpID) >= 2
ORDER BY   AvgSalary DESC;
⚠ Execution order: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. You cannot use a SELECT alias in HAVING — use the full expression instead.
Quick mental model
-- WHERE  = filter BEFORE grouping (individual rows)
-- HAVING = filter AFTER  grouping (aggregated groups)

SELECT DeptID, COUNT(*) AS Total
FROM   Employees
WHERE  Salary > 40000        -- removes low-earner rows first
GROUP BY DeptID
HAVING COUNT(*) >= 2;        -- then keeps groups with 2+ members

NULL Value Handling
IS NULL, IS NOT NULL, COALESCE, ISNULL / IFNULL, NULLIF
NULL Safety
IS NULL / IS NOT NULL — both DBs
-- Find employees with no department assigned
SELECT EmpName, DeptID
FROM   Employees
WHERE  DeptID IS NULL;

-- Find employees WITH a department
SELECT EmpName, DeptID
FROM   Employees
WHERE  DeptID IS NOT NULL
ORDER BY DeptID;
⚠ Never use = NULL or != NULL — these always return no rows. NULL comparisons require IS NULL / IS NOT NULL.
COALESCE — standard for both DBs
-- Return first non-NULL value from list
SELECT
    EmpName,
    COALESCE(DeptID, 0)       AS SafeDeptID,
    COALESCE(
        CAST(DeptID AS VARCHAR(10)),
        'Unassigned'
    )                          AS DeptLabel
FROM   Employees
ORDER BY DeptID;
✓ COALESCE works identically in SQL Server & MySQL
SQL Server — ISNULL()
-- Replace NULL DeptID with 0
SELECT
    EmpName,
    ISNULL(DeptID, 0)     AS DeptID,
    ISNULL(
        CAST(DeptID AS NVARCHAR(10)),
        N'Unassigned'
    )                      AS DeptLabel
FROM   Employees;

-- NULLIF: return NULL if two values are equal
SELECT
    EmpName,
    NULLIF(DeptID, 0) AS DeptOrNull
FROM   Employees;
🔵 SQL Server ISNULL(col, replacement) — accepts only 2 arguments (use COALESCE for multiple fallbacks).
MySQL — IFNULL() & NULLIF()
-- Replace NULL DeptID with 0
SELECT
    EmpName,
    IFNULL(DeptID, 0)     AS DeptID,
    IFNULL(
        CAST(DeptID AS CHAR),
        'Unassigned'
    )                      AS DeptLabel
FROM   Employees;

-- NULLIF: return NULL if two values are equal
SELECT
    EmpName,
    NULLIF(DeptID, 0) AS DeptOrNull
FROM   Employees;
🟠 MySQL ISNULL(col) only returns 1 or 0 — it does NOT replace the value. Use IFNULL(col, val) instead.
COUNT(*) vs COUNT(col) — NULL behaviour (both DBs)
-- COUNT(*) counts all rows; COUNT(col) ignores NULLs in that column
SELECT
    COUNT(*)       AS TotalRows,      -- 15 (includes Zahir)
    COUNT(DeptID)  AS WithDept,       -- 14 (Zahir excluded — DeptID is NULL)
    COUNT(DISTINCT DeptID) AS UniqueDepts -- 5  (NULL not counted as distinct)
FROM   Employees;
✓ Key rule: aggregate functions (SUM, AVG, MIN, MAX, COUNT(col)) silently skip NULL rows

Row-level if / else — evaluated per row
Salary Band + Tenure
SQL Server — date comparison with string literal
SELECT
    e.EmpName, d.DeptName, e.Salary,
    CASE
        WHEN e.Salary >= 75000 THEN 'Senior'
        WHEN e.Salary >= 55000 THEN 'Mid-Level'
        WHEN e.Salary >= 40000 THEN 'Junior'

    END AS Tenure
FROM       Employees e

ORDER BY   e.Salary DESC;

SELECT
    e.EmpName, d.DeptName, e.Salary,

IN & NOT IN
Match against a value list or subquery result
IN — value list (both DBs)
-- Employees in Eng, HR, Finance
SELECT EmpName, DeptID, Salary
FROM     Employees
WHERE    DeptID IN (1, 2, 3)
ORDER BY DeptID, Salary DESC;
NOT IN — value list (both DBs)
-- Exclude Engineering & HR
SELECT EmpName, DeptID, Salary
FROM     Employees
WHERE    DeptID NOT IN (1, 2)
  AND    DeptID IS NOT NULL
ORDER BY Salary DESC;
IN — subquery: Dhaka depts (both DBs)
SELECT EmpName, Salary
FROM   Employees
WHERE  DeptID IN (
    SELECT DeptID
    FROM   Departments
    WHERE  Location = 'Dhaka'
)
ORDER BY Salary DESC;
NOT IN — subquery: empty depts (both DBs)
SELECT DeptName, Location
FROM   Departments
WHERE  DeptID NOT IN (
    SELECT DeptID
    FROM   Employees
    WHERE  DeptID IS NOT NULL
);
-- Result: Legal (no employees)
⚠ NULL trap (both DBs): if the subquery returns even one NULL, NOT IN returns zero rows. Always add WHERE col IS NOT NULL inside the subquery — or use NOT EXISTS.
